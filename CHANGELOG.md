### Fixed
- Fresh sandbox images no longer leak root-owned uv tool paths into the default `sandbox` user's runtime. DeepAgents still installs into image-level paths at build time, but runtime `UV_TOOL_DIR`/`UV_TOOL_BIN_DIR` now point at sandbox-owned home paths, and the optional Hermes install directory is chowned to `sandbox` so `uv pip install --python /usr/local/lib/hermes-agent/venv/bin/python ...` works without sudo.
- Ad-hoc `uv pip install` into the Hermes venv no longer fails with `Permission denied` after a UID sync. The Dockerfile chowns `/usr/local/lib/hermes-agent` to the build-time sandbox UID, but the entrypoint remaps `sandbox` to the host UID on boot and its re-chown only covered `/home/sandbox`, leaving the venv (a system path) orphaned. Requires an image rebuild ([#364](https://github.com/mifunedev/openharness/issues/364)).
- Secret-exposure hooks (`.claude/hooks/{deny-env-dump,deny-secret-paths}.sh`) no longer deny operations on tracked template env files (`.example.env`, `.env.example`, `.env.sample`, `.env.template`); real `.env`, `.env.local`, `.env.production` continue to deny. Codex wrapper inherits the fix ([#356](https://github.com/mifunedev/openharness/issues/356)).
