## Introduction

Adopt Andrej Karpathy's [LLM Wiki pattern](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) (April 2026 gist) — a personal-scale knowledge base **compiled and maintained by an LLM** in markdown, loaded directly into context rather than retrieved via vector embeddings. Three layers (`raw/` source captures, `wiki/` entity pages, schema doc) and three operations (ingest, query, lint), grafted onto Open Harness conventions without duplicating existing memory / rules / docs surfaces.

The wiki holds **facts and synthesis** about topics the harness keeps re-deriving — distinct from `context/rules/*` (behavioral norms), `memory/MEMORY.md` (experiential journal), `docs/` (human-facing prose), and `.claude/skills/*` (executable procedures). The plan's § *Boundary table* is the load-bearing arbiter; every story's acceptance criteria preserve those boundaries.

## Goals

- Establish a `wiki/` top-level directory governed by `context/rules/wiki.md` as the schema doc (auto-loaded with other rules; no `CLAUDE.md` symlink collision)
- Persist `WebFetch` snapshots at `wiki/raw/<yyyy-mm-dd>-<slug>.md` (fetch-as-raw) with provenance trail to every entity page that cites them
- Ship three skills (`/wiki-ingest`, `/wiki-query`, `/wiki-lint`) matching Karpathy's three operations, each with a frozen argument interface and `argument-hint` frontmatter
- Enforce the orchestrator-only write boundary: sub-agents draft to `memory/<today>/wiki-drafts/`; orchestrator promotes via `/wiki-ingest --from-draft`
- Make every entry < 600 words, frontmatter-typed, `[[slug]]`-linked, with at least one `sources:` reference
- Manual lint cadence only (no heartbeat cron in first cut)

- [ ] `.gitignore` contains EXACTLY two lines for the raw exemption, **appended at the end of `.gitignore`**: `wiki/raw/*` then `!wiki/raw/README.md`. The bare-directory form `wiki/raw/` would silently ignore the README (lesson from PR #347's `.claude/plans/*` parallel). Verify with `git check-ignore -v wiki/raw/README.md` returning no match (exit code 1) and `git check-ignore -v wiki/raw/2026-05-24-test.md` returning a match (exit code 0).
- [ ] `context/rules/wiki.md` ships **complete** with ALL of the following required sections, each non-empty (a stub of just the headings does NOT satisfy "complete"):
  1. **Boundary table** — the 7-row table from the plan § *Boundary table*, reproduced verbatim
  2. **Entry schema** — the full frontmatter spec (title, slug, tags, created, updated, sources, related, confidence), the H1 + `## Summary` + `## Detail` + `## See Also` body layout, ≤600-word cap, sub-article split convention (`<parent>/<child>.md`)

  4. **Cross-link convention** — `[[slug]]` syntax (Obsidian-style); grep pattern: `grep -roE '\[\[[a-z0-9-]+\]\]' wiki/`
  5. **Confidence lifecycle** — `provisional` set by `/wiki-ingest` on create; `confirmed` set MANUALLY by the orchestrator (e.g., via `Edit`); `deprecated` set MANUALLY by the orchestrator (never autonomously by `/wiki-lint`); `/wiki-lint` REPORTS entries with `confidence: deprecated` but does NOT set the flag
  6. **Frontmatter extraction canonical command** — both `/wiki-query` and `/wiki-lint` MUST extract YAML frontmatter using: `awk '/^---$/{f=!f; next} f{print}' wiki/<slug>.md`. This is the locked extraction method; deviation creates silent divergence between the two skills.
  7. **Body-merge strategy for `/wiki-ingest` updates** — when re-ingesting a source that matches an existing entry's slug: REPLACE the `## Summary` and `## Detail` sections in-place; APPEND the new snapshot path to the `sources:` frontmatter list; APPEND new `[[link]]` candidates to `## See Also` (deduplicated); update `updated:` to today UTC. Do NOT concatenate bodies. Do NOT touch `created:`.

- [ ] `wiki/README.md` table format documented so `/wiki-lint` regeneration in US-004 can match exactly: **exact column header line** `| Slug | Title | Tags | Updated |`, alignment row `| --- | --- | --- | --- |`, sorted by `updated:` descending. An empty corpus produces the two header lines only — this is NOT an error condition.

### US-002: `/wiki-ingest` skill

**Description:** As the orchestrator, I want a skill that reads a source URL/path/draft, snapshots it to `wiki/raw/`, and writes or updates entity pages in `wiki/`.

**Acceptance Criteria:**

- [ ] `.claude/skills/wiki-ingest/SKILL.md` exists with valid frontmatter including `name: wiki-ingest`, `description`, and `argument-hint: "<url|path> [--slug <override>] | --from-draft <slug> [--allow-stale]"`
- [ ] Argument interface is locked: `/wiki-ingest <url|path> [--slug <override>]` for source ingest, `/wiki-ingest --from-draft <slug> [--allow-stale]` for draft promotion. No other invocation forms.
- [ ] Skill MUST `mkdir -p wiki/raw/` before writing any snapshot (since `wiki/raw/` is gitignored — the directory may not exist on a fresh clone)
- [ ] Slug derivation: read the rule defined in `context/rules/wiki.md` § *Slug derivation rule* (US-001). URL ingest derives slug from URL path's last non-UUID segment, lowercased + kebab-case; if path is a bare UUID/hash (e.g., gist URLs), `--slug <override>` is required and the skill errors if absent.
- [ ] For URL ingest: WebFetch the URL, write snapshot to `wiki/raw/<yyyy-mm-dd>-<slug>.md` with header `# Source: <url>\n\n<fetched body>`. Date is `date -u +%Y-%m-%d` (UTC).
- [ ] For path ingest: read the file, write a snapshot copy to `wiki/raw/<yyyy-mm-dd>-<basename>.md`. Same UTC date convention. `--slug <override>` is optional; defaults to slugified `<basename>` per the schema rule.
- [ ] For `--from-draft <slug>`: resolve the draft via glob `memory/*/wiki-drafts/<slug>.md` (NOT today's date hardcoded). Sort matches by the **ISO date component in the parent directory name** (`memory/YYYY-MM-DD/`); take the lexicographically greatest date. Do NOT use filesystem mtime (unreliable across git checkout / docker volume mounts).
- [ ] **Stale draft handling**: if the most-recent draft directory date is more than 7 days older than today (UTC), the skill exits with a `STALE` message unless `--allow-stale` is passed. Drafts can also be suppressed by renaming `<slug>.md` to `<slug>.md.skip` — the glob excludes `.skip` files.
- [ ] Write or update `wiki/<slug>.md` with valid frontmatter per `context/rules/wiki.md` schema: `title`, `slug`, `tags`, `created` (only on create), `updated` (always), `sources` (appending the new snapshot path), `related` (optional), `confidence: provisional` (on create)
- [ ] **Update path is in scope**: when an existing `wiki/<slug>.md` matches the ingest source's slug, the skill MUST update that page using the **body-merge strategy locked in `context/rules/wiki.md` § *Body-merge strategy***: replace `## Summary` + `## Detail` in-place; append new snapshot to `sources:`; append deduplicated `[[link]]` candidates to `## See Also`; update `updated:`; do NOT touch `created:`; do NOT concatenate bodies.
- [ ] Daily log entry per `context/rules/memory.md` Memory Improvement Protocol: `## /wiki-ingest -- HH:MM UTC` heading, fields `Source` (URL or path), `Slug-Created` (slug or `—`), `Slugs-Updated` (comma list or `—`), `Snapshot-Path`, `Result`, `Observation`
- [ ] CHANGELOG `[Unreleased]` `### Added` entry: "`/wiki-ingest` skill for capturing WebFetch snapshots and authoring wiki entries"
- [ ] **Smoke test (manual QA only — not run in CI; CI uses a local fixture if it tests this skill)**: invoking `/wiki-ingest https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f --slug karpathy-llm-wiki` creates a snapshot at `wiki/raw/<today>-karpathy-llm-wiki.md` AND an entity page at `wiki/karpathy-llm-wiki.md` with valid frontmatter, `confidence: provisional`, and the snapshot path in `sources:`. **This smoke test MUST run AND commit before US-003's smoke test runs.**

### US-003: `/wiki-query` skill

**Description:** As the orchestrator (or any session), I want a skill that searches the wiki by topic and reads the top match(es) into context.

**Acceptance Criteria:**

- [ ] `.claude/skills/wiki-query/SKILL.md` exists with valid frontmatter including `name: wiki-query`, `description`, and `argument-hint: "<topic>"`. **Multi-word topic semantics**: argument is split on whitespace into space-separated OR terms; an entry matches if ANY term appears in `title`, `slug`, or `tags` (union of per-term matches, deduplicated). Document this in the skill body.
- [ ] Greps only the **frontmatter** of each `wiki/*.md` entry using the canonical extraction command locked in `context/rules/wiki.md` § *Frontmatter extraction canonical command* (`awk '/^---$/{f=!f; next} f{print}' wiki/<slug>.md`) — NOT the entry body. Matches against `title`, `slug`, and `tags`.
- [ ] Returns a list of all matching `wiki/*.md` file paths to stdout
- [ ] **Empty-result behavior**: if `Match-Count = 0`, the skill prints `No wiki entries matched <topic>` and exits with status 0. This is NOT an error condition.
- [ ] **Sub-cap behavior**: if `0 < Match-Count < 3`, read ALL matches into context. If `Match-Count >= 3`, read the top 3 only (cap is hardcoded to `3` in v1; raising the cap requires editing `SKILL.md`).
- [ ] Tie-breaking order is `updated:` descending (most recently updated first)
- [ ] Reads individual `wiki/*.md` files directly — does NOT route via `wiki/README.md` index (the README is a human-orientation index, not a query backend)
- [ ] Daily log entry: `## /wiki-query -- HH:MM UTC` heading, fields `Query`, `Match-Count`, `Read-Slugs` (slugs of files read into context, or `—` if zero), `Result`, `Observation`
- [ ] CHANGELOG `[Unreleased]` `### Added` entry: "`/wiki-query` skill for grep-based wiki search with top-3 read cap"
- [ ] **Smoke test (manual QA only)**: invoking `/wiki-query karpathy` after US-002's smoke test has committed `wiki/karpathy-llm-wiki.md` returns that entry's path and reads it into context. Also verify: `/wiki-query nonexistent-topic-xyz` prints the empty-result message and exits 0. **This smoke test runs AFTER US-002's smoke test commits, not concurrently.**

### US-004: `/wiki-lint` skill

**Description:** As the orchestrator, I want a manual health-check skill that surfaces stale/orphaned/broken entries and regenerates the `wiki/README.md` index.

**Acceptance Criteria:**

- [ ] `.claude/skills/wiki-lint/SKILL.md` exists with valid frontmatter including `name: wiki-lint`, `description`, and `argument-hint: "[--dry-run]"`
- [ ] Default invocation regenerates `wiki/README.md` (entries table). `--dry-run` prints what would change without writing.
- [ ] **Atomic regeneration**: write to `wiki/README.md.tmp` first; validate it is non-empty AND contains the exact header line `| Slug | Title | Tags | Updated |`; only on validation success, atomically rename to `wiki/README.md`. On validation failure, exit non-zero, leave the original `wiki/README.md` intact, and print the failure reason.
- [ ] **Stale check (a)**: enumerate entries where frontmatter `updated:` is > 90 days older than today (UTC). Print as separate finding type with recommendation "consider review".
- [ ] **Stale check (b)**: enumerate entries where frontmatter `confidence: deprecated`. Print as separate finding type with recommendation "consider archive or delete". Stale (a) and (b) MUST be reported separately, never conflated. `/wiki-lint` ONLY REPORTS deprecated entries — it never autonomously SETS `confidence: deprecated` (that flag is set manually by the orchestrator only, per the lifecycle in `context/rules/wiki.md`).
- [ ] **Orphan check**: two-pass. Pass 1: enumerate all entry slugs from `wiki/*.md` frontmatter using the canonical extraction command (`awk '/^---$/{f=!f; next} f{print}'`) — same as `/wiki-query` (US-003). Pass 2: grep all entry bodies for `[[slug]]` occurrences (pattern `\[\[[a-z0-9-]+\]\]` — slug charset is constrained in `context/rules/wiki.md`); build inbound-reference count per slug. Entries with zero inbound `[[links]]` are orphans. **Note**: a single-entry corpus is always an orphan by definition — this is a true positive, not a false positive.
- [ ] **Broken-outbound-link check**: grep all entry bodies for `[[slug]]` references where `slug` does NOT match any existing entry's frontmatter slug. Report separately from orphans.
- [ ] **Contradiction detection**: explicitly **descoped** for v1. The skill includes a stub function that prints `contradiction detection: not yet implemented — see wiki-lint follow-up tracking` (no issue # placeholder; tracking issue may be filed post-merge but is not blocking).
- [ ] **README regeneration format**: entries table is `| Slug | Title | Tags | Updated |` (literal column header, in that order), alignment row `| --- | --- | --- | --- |`, sorted by `updated:` descending. Empty corpus produces the two header lines and no data rows.
- [ ] Daily log entry: `## /wiki-lint -- HH:MM UTC` heading, fields `Entries-Scanned`, `Stale-90d`, `Deprecated`, `Orphaned`, `Broken-Links`, `Mode` (op/dry-run), `Result`, `Observation`
- [ ] CHANGELOG `[Unreleased]` `### Added` entry: "`/wiki-lint` skill for manual wiki health checks + index regeneration"
- [ ] **Smoke test (manual QA only)**: with a one-entry corpus (post US-002), invoking `/wiki-lint --dry-run` reports zero stale-90d findings and zero deprecated findings (the single entry IS an orphan, which is a true positive — the smoke test's "zero false positives" claim refers ONLY to the stale + deprecated + broken-link checks on a fresh entry, NOT the orphan check). Setting that entry's `confidence: deprecated` and re-running surfaces it under the deprecated check. Running without `--dry-run` regenerates `wiki/README.md` to include the entry under the header lines.

## Functional Requirements

- FR-1: Top-level `wiki/` directory MUST be tracked; `wiki/raw/*` MUST be gitignored except for `wiki/raw/README.md`
- FR-2: `context/rules/wiki.md` MUST be the sole schema document; no `wiki/CLAUDE.md` (would collide with the root `CLAUDE.md` symlink to `AGENTS.md`)
- FR-3: All wiki entry frontmatter MUST conform to the schema: `title`, `slug`, `tags`, `created`, `updated`, `sources`, `related` (optional), `confidence` (`provisional` | `confirmed` | `deprecated`)
- FR-4: Cross-links MUST use `[[slug]]` syntax (Obsidian-style, grep-friendly)
- FR-5: Every wiki entry MUST be ≤ 600 words; over-cap entries MUST be split into sub-articles (`<parent>/<child>.md`)
- FR-6: Every wiki entry MUST have at least one `sources:` reference
- FR-7: All three new skills MUST log to `memory/<today>/log.md` per `context/rules/memory.md` Memory Improvement Protocol
- FR-8: All three new skills MUST include `argument-hint` in their SKILL.md frontmatter
- FR-9: `/wiki-ingest` MUST default `confidence: provisional` on entry creation; orchestrator manually upgrades to `confirmed`; `/wiki-lint` flags `deprecated` candidates by heuristic
- FR-10: `/wiki-lint` MUST support `--dry-run` for consistency with `/context-audit` and `/reflect`
- FR-11: Sub-agents MUST NOT write directly to `wiki/`. They MAY write drafts to `memory/<today>/wiki-drafts/<slug>.md`; only the orchestrator promotes via `/wiki-ingest --from-draft <slug>`
- FR-12: Every PR landing US-001 through US-005 MUST include its own `CHANGELOG.md` `[Unreleased]` entry per `context/rules/git.md` § Changelog

## Non-Goals (Out of Scope)

- **NO vector DB / embeddings / RAG infrastructure** — grep + read-into-context is the entire retrieval mechanism. This is a deliberate stance matching Karpathy's gist and the harness's "short feedback loops" principle.
- **NO heartbeat cron for `/wiki-lint`** — manual invocation only in this first cut. Re-evaluate after first 90 days of usage.
- **NO automatic auto-loading of `wiki/README.md`** into every session — defer to `/context-audit` after the corpus has ≥ 10 entries.
- **NO bootstrap corpus pre-seeding** in this PRD (the plan's PR 5). First real `/wiki-ingest` runs seed the corpus organically.
- **NO contradiction detection** in `/wiki-lint` (US-004) — explicitly stubbed and deferred to a follow-up issue.
- **NO migration of existing `memory/<topic>.md` notes** into the wiki in this PRD — the boundary table in `context/rules/wiki.md` defines when a topic note graduates to a wiki entry ("re-derived twice and earns a schema"); promotion is opportunistic, not bulk.
- **NO `/reflect` → wiki handoff** in this PRD — defer the integration to a follow-up issue.
- **NO `CLAUDE.md`-style nested schema doc** inside `wiki/` — `context/rules/wiki.md` is the sole governance document.

## Design Considerations

- The `[[slug]]` cross-link syntax is grep-friendly (`grep -roE '\[\[[a-z0-9-]+\]\]' wiki/`) — no parser needed. `/wiki-lint`'s orphan + broken-link checks rely on this grep pattern.
- `wiki/README.md` follows `context/rules/directory-readme.md`: one-line intent, subfolder enumeration, conventions, pointer to canonical schema doc.
- Entry layout: H1 title, `## Summary` (2-3 sentences), `## Detail` (≤ 600 words), `## See Also` (`[[slug]]` bullets).

## Technical Considerations

- All three skills are pure orchestrator-side scripts (markdown read/write + grep + WebFetch). No application code, no in-sandbox execution.
- `wiki/raw/` snapshots are immutable once written. Re-ingesting an already-captured URL appends a new dated snapshot rather than overwriting.
- Concurrency model inherits from `context/rules/memory.md`: only the orchestrator writes to tracked surfaces. Sub-agent drafts are a side channel.
- The `/ralph` runner converts this PRD to `tasks/karpathy-llm-wiki/prd.json`; `/delegate` executes the user stories as parallel waves where deps permit (US-002 + US-003 can land in parallel after US-001; US-004 depends on both; US-005 depends on all of US-002 through US-004).

## Success Metrics

- After the smoke test sequence (US-002 → US-003 → US-004 in order), `/wiki-query karpathy` reads the Karpathy entry into context and `/wiki-lint --dry-run` reports zero false positives
- `git status` after each PR shows ONLY the files that PR was scoped to touch (no leakage into `memory/MEMORY.md`, `context/IDENTITY.md`, or other surfaces)
- Boundary integrity: zero content in `wiki/*.md` duplicates content in `context/rules/*.md`, `context/IDENTITY.md`, or `memory/MEMORY.md` (manual reviewer check on first three entries)
- A future session asking about a previously-ingested topic gets the answer from a wiki entry rather than re-deriving it from scratch

## Open Questions

- `/reflect` integration: when `/reflect` surfaces a topic note that has been re-derived twice (per boundary table), should it auto-suggest `/wiki-ingest`, or just flag in the log? Decision deferred; file as a follow-up issue after first 10 wiki entries land.
- Auto-load decision for `wiki/README.md`: at what corpus size does the index become valuable enough to surface in every session vs. its token cost? `/context-audit` re-evaluates after ≥ 10 entries.
- Deprecated entry lifecycle: `confidence: deprecated` is defined, but the recommended action (archive to `wiki/archive/<slug>.md` vs. hard delete) is undecided. Define in `context/rules/wiki.md` after first deprecation in practice.
- Contradiction detection heuristic (deferred from US-004): what's the actual algorithm? Possibilities include LLM-based diff of overlapping claims, frontmatter `related:` cross-check, or human-curated flag. To be designed in follow-up issue.
