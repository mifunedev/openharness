## Codebase Patterns

- **croner v9.1.0 has no `Cron.validate`** (`typeof Cron.validate === "undefined"`). Validate a schedule by constructing `new Cron(schedule)` with NO callback and NO `name` option inside try/catch: it parses synchronously and throws on a bad pattern, arms no timer (croner only calls `schedule()` when a function is passed), and registers nothing in the module-level named-jobs array (no `name`). `.stop()` is defensive only.
- **Verified probe outcomes**: `''` throws, `'not-a-cron'` throws, `'0 * * * *'` constructs OK. So `isValidSchedule('')===false`, `isValidSchedule('not-a-cron')===false`, `isValidSchedule('0 * * * *')===true`.

- **Injectable-logFn seam**: `onJobError(id, err, logFn = log)` is the established pattern for test-injecting the logger. US-002 must mirror it: `loadCrons(dir?, logFn = log)`.
- **`pnpm run test:scripts`** is the authoritative gate (runs all 13 vitest files, currently 151 tests). `pnpm run typecheck` and `pnpm run lint` both pass clean.

US-003's `2 scheduled, 1 skipped` BOOT count is the count of crons present-but-dropped; the simplest accurate source is comparing total `*.md`/enabled crons found vs `loadCrons` output length, or having `loadCrons` surface a skip count.
