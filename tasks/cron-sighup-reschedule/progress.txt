# progress

## Codebase Patterns

- **No tsconfig at root.** `scripts/*.ts` is transpiled by vitest/esbuild (type
  stripping, no type-check) and editor TS. The root `pnpm run typecheck` only
  covers `packages/**` + `packages/oh`. `@types/node` is NOT installed, so an
  isolated `tsc` on `scripts/cron-runtime.ts` fails to resolve `node:*` imports.
  To genuinely type-check a snippet, drop a temp `.ts` under `scripts/`, run the
  local `./node_modules/.bin/tsc --noEmit --strict --skipLibCheck
  --moduleResolution bundler --module esnext --target es2022 <file>` (avoid
  `--types node`), then delete it. The real CI gate for scripts is
  `pnpm run test:scripts` (vitest, 14 files / 186 tests).
- **`Cron | void` narrows cleanly.** `if (handle) activeJobs.push(handle)` where
  `handle: Cron | void` narrows to `Cron` under `--strict` — no cast needed.
- **Test-injection seams convention.** `loadCrons(dir, logFn=log)`,
  `onJobError(id, err, logFn=log)`, and now `scheduleAll(dir, logFn, mkCron)`
  expose optional last params that default to module-private impls purely so
  tests can drive behavior without real side effects. New seams (e.g.
  `resetActiveJobs()`) should carry a "no external-stability guarantee" comment
  to match the existing ones.
- **Logging is `log(id, status, msg)`** → one tab-separated line to
  `crons/.cron.log` via `appendFileSync`. Tests mock only `appendFileSync`
  (vi.mock("node:fs") pass-through) to observe status tokens. Reuse the private
  `log()` helper for any new token (US-003 RELOAD) — never inline appendFileSync.
- **`scheduleAll` counting is disjoint:** `loadSkips` (invalid at load) +
  `constructSkips` (throws at `mkCron`); `scheduled++` fires on every successful
  `mkCron` regardless of handle truthiness, so a `() => void` test spy still
  counts as scheduled.

## US-003 — 2026-06-13 12:35 UTC

**Title**: Reload is fault-isolated and logs a RELOAD liveness line
**Files changed**: scripts/cron-runtime.ts
**Commit**: 7e7e430
**Result**: PASS

### What I did
Captured `scheduleAll()`'s `BootResult` in `sighupHandler` as `const { scheduled,
skipped } = scheduleAll()` and emit one `log("system", "RELOAD", `${scheduled}
scheduled, ${skipped} skipped`)` line on a successful reschedule — reusing the
private `log()` helper (no inline appendFileSync, no duplicated format), id
"system" matching the BOOT precedent. Fault-isolation (AC #1) required no code:
`scheduleAll()` already routes through `loadCrons()`, which drops invalid-schedule
files as SCHED_INVALID at reload time exactly as at boot, so a malformed file
present during a SIGHUP can't crash the reload. Expanded the handler comment to
document the RELOAD line and the fault-isolation guarantee.

### Learnings for future iterations
- **The RELOAD log is the US-004 assertion target**: it is the only line written
  by `sighupHandler` itself (everything else comes from `scheduleAll`/`loadCrons`).
  US-004's "asserts a RELOAD line is written via the existing appendFileSync spy"
  AC should grep the spy's captured lines for a `\tRELOAD\t` token after calling
  `sighupHandler()` — but note: `sighupHandler` calls the *real* module-private
  `log()` (not an injected `logFn`), so the test must spy on `appendFileSync`
  (the vi.mock("node:fs") pass-through pattern), NOT pass a fake logFn. The BOOT
  line from the nested `scheduleAll()` is also written through the same `log()`,
  so assert on RELOAD specifically, not just "any appendFileSync call".
- **RELOAD only fires on success**: the `log()` call sits AFTER `scheduleAll()`
  inside the try, so if `scheduleAll()` ever throws, the finally still clears the
  `reloading` lock but no RELOAD line is written — matching the "on a successful
  reschedule" AC. US-004's empty-registry and re-entrancy tests should expect
  RELOAD to appear once per *completed* handler call (the re-entrant no-op path
  returns before reaching it).
- **US-004 is now the only remaining code story**; US-005 is docs/CHANGELOG only.
  The implementation surface (US-001..US-003) is complete and green at 186 tests.

## US-002 — 2026-06-13 06:33 UTC

**Title**: Exported, testable SIGHUP handler reschedules without exiting
**Files changed**: scripts/cron-runtime.ts
**Commit**: a542e2a
**Result**: PASS

### What I did
Added an exported `sighupHandler()` that stops every handle in the module-level
`activeJobs` registry via `.stop()` (each wrapped in try/catch so one stubborn
handle can't abort the reload), then calls `scheduleAll()` to re-read crons/ and
re-arm. Guarded it with a module-level `reloading` boolean lock cleared in a
`finally` so a rapid double-SIGHUP is a safe no-op. Registered it in `main()` via
`process.on("SIGHUP", () => setImmediate(sighupHandler))` (deferred off the
signal-callback context); SIGTERM/SIGINT cleanup is untouched. The handler never
calls process.exit() and never unlinks the PID file.

### Learnings for future iterations
- **US-003 hook point**: `sighupHandler` currently calls `scheduleAll()` and
  discards its `BootResult`. US-003 should capture `const { scheduled, skipped }
  = scheduleAll()` and add `log("system", "RELOAD", \`${scheduled} scheduled,
  ${skipped} skipped\`)` after it. This is the one explicit modification to
  US-002's completed body that a later story is allowed to make.
- **US-004 re-entrancy test seam**: the `reloading` lock is module-private. To
  test the "second invocation mid-reload is a no-op" AC, drive re-entry from
  inside the reload — e.g. inject an `mkCron` (via the scheduleAll the handler
  calls is not directly injectable, so) instead make the FIRST `activeJobs`
  handle's `.stop()` spy call `sighupHandler()` again, then assert scheduleAll
  ran once. Handles are seeded into `activeJobs` by calling `scheduleAll(tmp,
  spy, () => fakeHandleWithStopSpy)` before invoking `sighupHandler()`.
- **Type-check caveat (confirmed)**: standalone `tsc` on cron-runtime.ts reports
  only the documented missing-`@types/node` class (`process`, `setImmediate`,
  `node:*`). My new `setImmediate` usage adds one such line — expected, not a
  real error. `pnpm run test:scripts` (186 tests) is the true gate and passed.
