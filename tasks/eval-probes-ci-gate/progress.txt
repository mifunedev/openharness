## Codebase Patterns

- **YAML parse check in orchestrator env**: the `python3 -c "import yaml"` parse
  check from the ACs FAILS here â€” the orchestrator's python3 has no `yaml` module
  and `pip`/`ruby`/`yq` are absent. Equivalent check: js-yaml is in the pnpm store,
  require it by full path:
  `node -e "const y=require('./node_modules/.pnpm/js-yaml@4.2.0/node_modules/js-yaml');const fs=require('fs');y.load(fs.readFileSync('<file>','utf8'))"`.
  Note js-yaml 4.x uses YAML 1.2, so the `on:` key parses as the STRING `"on"`,
  NOT boolean `true` (the YAML-1.1 footgun) â€” access `doc['on'].push.paths`.
- **ci-harness.yml path lists are duplicated** verbatim across `push.paths` and
  `pull_request.paths`. To Edit one block at a time, anchor on the differing
  trailing line: push block ends `â€¦workspace/**"\n  pull_request:`; pull_request
  block ends `â€¦workspace/**"\n\nconcurrency:`.
- **PCRE-support detection idiom**: the literal AC snippet `grep -P . /dev/null`
  is WRONG â€” an empty file yields exit 1 even when `-P` IS compiled in, so it
  always-skips. Use `printf 'x\n' | grep -qP 'x' 2>/dev/null` (exit 0 on real
  PCRE support; non-zero only when `-P` is genuinely absent). Sandbox + ubuntu-latest
  GNU grep both support `-P`, so owned-surface-guard PASSes in CI; the guard is
  belt-and-braces for a minimal runner.
- **Hermeticity demo via PATH shim**: to prove a probe SKIPs (not ERRORs) when a
  tool is missing, run it under a restricted `PATH` (coreutils-only, omit
  pgrep/tmux/zsh) or a `grep` shim that rejects `-P`. CAVEAT: a naive `-P` shim
  that scans ALL args matches the *pattern* `-- "${OWNED_PATHS[@]}"` (starts `-`,
  contains `P`) as if it were a `-P` flag â€” a faithful shim must `break` at `--`
  and only reject option args. Real GNU grep ends options at `--`, so the probe
  itself is fine; the over-broad shim was a test artifact.
- **Most probes are static-grep over committed files** â†’ inherently hermetic
  (PASS in CI; SKIP only if their target file is absent, which never happens in a
  checkout). Only probes touching live deps (process-inspection: next-dev-prod;
  extract-and-run: drift-check-cron-staleness-glob; tool-dep: owned-surface-guard)
  need cold-runner scrutiny. See `probe-audit.md` for the full 16-probe table.
- **No probe uses node/jq/python/ruby** â€” confirmed for US-003: checkout + bash
  suffices, no tooling-setup step needed.
- **CI job RESULTS.md churn**: the runner rewrites `evals/RESULTS.md` (timestamps)
  on every invocation. After running the suite locally to verify a story, `git
  checkout -- evals/RESULTS.md` to drop that churn so it never sneaks into a
  non-US-002 commit. The CI `eval-probes` job mirrors this â€” it gates on exit
  code only and has no `git add/commit/push` of RESULTS.md (AC #4).
- **Mirror boot-lint for independent jobs**: a self-contained probe/lint job that
  needs no toolchain is just `runs-on: ubuntu-latest` + `actions/checkout@v4` +
  one `run:` step. No per-job `permissions:` block â†’ it inherits the workflow's
  top-level `permissions: contents: read`. boot-lint is the template; eval-probes
  follows it.
- **A new probe is INERT until its PASS row is in the committed RESULTS.md**:
  `run.sh:106` only flags a regression on `prior == "PASS"` (prior = committed
  baseline status). A probe with no committed row â†’ `prior` empty â†’ a red run is
  recorded as `new-fail` delta but NEVER added to `regressions[]` â†’ suite still
  exits 0. So adding a self-guard probe MUST also add its `PASS` row to the
  committed baseline, or the guard can't fire. The committed RESULTS.md already
  holds one row per probe (invariant: 16/16 before US-004), so this also keeps
  that invariant intact.
- **Add a single baseline row WITHOUT full-suite churn**: run the suite once to
  verify it integrates + capture the runner's exact row
  (`grep '^| <id> ' evals/RESULTS.md`), then `git checkout -- evals/RESULTS.md`
  to drop the 16 timestamp bumps, and `Edit`-insert just that one row in its
  glob-sorted position. The runner glob-sorts probes in C-locale order, so
  `eval-ci-gate` sorts BEFORE `eval-gate` (`c` < `g`) â€” insert between
  `drift-check-cron-staleness-glob` and `eval-gate`. Result: a 1-line diff.
- **Verify a 3-state probe's branches in an isolated temp tree, never by editing
  the real workflow**: build `$TMP/{evals/probes,.github/workflows}`, copy the
  probe in, and feed it a workflow variant (`grep -v 'runner line'` for the
  REGRESSION case, `rm` it for the SKIPPED case). The probe resolves
  `WORKFLOW` via `${BASH_SOURCE[0]}/../../.github/...`, so running the COPY makes
  it read the temp workflow â€” zero risk to the committed `ci-harness.yml`.
