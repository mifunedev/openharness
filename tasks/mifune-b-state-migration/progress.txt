- Dir-move mechanics (proven US-001, reuse for US-002 hooks): `git mv
  .claude/<x> .mifune/<x>` moves the whole dir+contents (git records clean
  renames), then `ln -s ../.mifune/<x> .claude/<x>` + `git add .claude/<x>`
  registers the back-symlink as a git symlink (mode 120000), exactly like
  `.claude/skills`. NO probe couples to the agents path, so the move was
  eval-neutral (all 56 PASS/SKIPPED). Verify the symlink mode with
  `git ls-files -s .claude/<x>` (expect `120000`).
- Roadmap ticking is owned by US-006 ("ticked through M6"); per-story

- CHANGELOG: agents move went under `### Changed` (a relocation), right after
  the analogous skills-move bullet; referenced roadmap issue #302.
- CI path filters DON'T follow symlinks (US-002): GitHub Actions `paths:`
  matches the literal git path of changed files, so after moving a dir to
  `.mifune/<x>` a future edit lands at `.mifune/<x>/...` and would NOT match a
  `.claude/<x>/**` filter (the symlink only redirects path RESOLUTION, not diff
  matching). The static `harness-ci-hooks-paths` probe greps the workflow for
  the literal `.claude/hooks/**` (×2) + `.claude/hooks/*.sh` strings, so it
  can't catch this runtime gap. Fix = ADD `.mifune/<x>/**` to both push +
  pull_request filters ALONGSIDE the probe-pinned `.claude/<x>/**` (don't
  remove it — the probe needs it; the shellcheck glob `.claude/hooks/*.sh`
  still lints the real files through the symlink). This mirrors the skills
  precedent (the CI filter is `.mifune/skills/**`, no `.claude/skills/**`).
  Future US-003 (scripts→skill dirs) has the same trap if any moved path is in
  a CI filter.
- Script-move root resolution (proven US-003, reuse for US-004 .oh/ relocations):
  a root-anchored script (one that reads `harness.yaml` or execs sibling
  `scripts/*.sh`) BREAKS when moved out of `scripts/` because it computes its
  root as `REPO_ROOT="$(dirname "$SCRIPT_DIR")"` (assumes `<root>/scripts/`).
  Fix = adopt the `.mifune/skills/eval/run.sh` walk-up idiom: `REPO_ROOT=$SCRIPT_DIR;
  while [ "$REPO_ROOT" != "/" ] && [ ! -f "$REPO_ROOT/scripts/cron-runtime.ts" ]; do
  REPO_ROOT="$(dirname "$REPO_ROOT")"; done`, then reference shared scripts as
  `$REPO_ROOT/scripts/<name>`. Verify with a gh-stub run from the NEW path
  (no "missing scripts/locked-append.sh" fallback warning = root resolved).
- CI coverage gaps from a scripts→skill move are TWO-fold and the eval suite
  catches NEITHER (CI-only): (1) the path FILTER — `.mifune/skills/**` was
  already in ci-harness.yml from the skills move, so moved skill scripts still
  trigger CI without a filter edit (unlike US-002 hooks). (2) the shellcheck
  GLOB — `scripts/*.sh` no longer covers the moved scripts; add
  `.claude/skills/<skill>/*.sh` to the glob (mirrors the `.claude/hooks/*.sh`
  precedent; shellcheck reads through the `.claude/skills -> ../.mifune/skills`
  symlink). Run the exact CI shellcheck line locally to confirm clean.
- The cron `preflight:` value must pass `isValidAgentBin` in cron-runtime.ts
  (`/^[A-Za-z0-9_./-]+$/`, no leading `-`, no `..`); `.mifune/skills/.../x.sh`
  passes. runPreflight resolves it via `path.resolve(process.cwd(), scriptPath)`
  at the root checkout (preflight runs BEFORE the worktree spawn), so a
  repo-relative `.mifune/...` path resolves correctly.
- Test/probe FIXTURE strings (drift-check-cron-staleness-glob.sh `bbb-valid.md`

- Rule→skill move mechanics (proven US-005, reuse for US-006 memory.md): `git mv
