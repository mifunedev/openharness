# Progress — oh-devcontainer-restructure (RFC #531 Phase 2 slice 2)

## Codebase Patterns

- Build context for the sandbox image is the REPO ROOT (compose `build.context`), so Dockerfile
  COPY sources are root-relative and mostly DO NOT change when the Dockerfile file itself moves.
  Only `COPY .devcontainer/entrypoint.sh` (a moved asset) changes.
- compose relative paths (`context`, `volumes` bind source) resolve from the compose FILE's dir.
  Moving the compose file from `.devcontainer/` to `.oh/devcontainer/` deepens it one level, so
  `..` becomes `../..`.
- The env files (`.env`, `.example.env`, `.harness.yaml.env`) are USER-OWNED and STAY at root
  `.devcontainer/`. Only the six build assets move. Repointing an env path is a bug.
- `.oh/**` is already a CI path filter in sandbox-boot-guard.yml, so the moved location is
  already covered; keep `.devcontainer/**` too (compat layer remains).
- `boot-lint-glob.sh` asserts the shellcheck glob covers active boot-script dirs — after the move
  the active dir is `.oh/devcontainer/` not `.devcontainer/`; the probe and BOTH workflows
  (ci-harness.yml + release.yml) must change together.
- Some vitest files reference moved assets via the SPLIT-FORM `path.join(ROOT, ".devcontainer",
  "<asset>")` — invisible to a `/`-literal grep. The full split-form set is enumerated in US-003.
  Validate with `node_modules/.bin/vitest run --root <worktree>` (the worktree has no node_modules;
  resolution walks up to the repo-root node_modules).
