# PRD — `oh update` (RFC #531 Phase 3)

> Upstream RFC: https://github.com/mifunedev/openharness/issues/531 — "Make
> OpenHarness portable via project-local `.oh` infrastructure". This is **Phase 3**:
> *"`oh update` upgrades only `.oh` infrastructure. Project source remains untouched."*

## Summary

Add an `oh update` subcommand to the in-sandbox `oh` CLI. In a repository that has
been *equipped* with OpenHarness (an `.oh/` control plane vendored in), `oh update`
refreshes **only** that `.oh/` control plane from a newer source, leaving every
project-owned file (`harness.yaml`, `AGENTS.md`, `src/`, `.devcontainer/.env`, …)
byte-for-byte untouched. It is the sibling of `oh init`: where `init` *seeds
project-owned files from `.oh/templates/`*, `update` *refreshes the `.oh/`
infrastructure itself* — the opposite side of the equip relationship.

## Why this is shippable standalone (base = `origin/development`)

`oh init` (PR #334, still open) writes **project files** scaffolded *from*
`.oh/templates/`. `oh update` writes the **`.oh/` control plane** (cli, scripts,
orchestration) — which already exists on `origin/development`. The two commands
touch **disjoint targets**, so Phase 3 does **not** depend on #334 being merged.
The only overlap is the one-line `cli.ts` subcommand registration (an `if (first
=== "update")` branch adjacent to where #334 adds `if (first === "init")`) — a
trivial merge union, the analog of the CHANGELOG/RESULTS unions slices 1 and 2
already carried. This continues the established slice-1/slice-2 pattern: each
slice is a standalone PR off `origin/development`, mergeable in any order.

## Design

### Source of the new control plane

`oh update --from <dir>` points at a built OpenHarness checkout; its `<dir>/.oh/`
is the new control plane. This mirrors **the exact precedent set by `oh init`**,
whose own PR (#334) defers installed-binary template bundling and requires an
explicit `--templates <dir>` until `/opt` bundling lands. Remote-fetch
(`oh update` pulling a ref over the network) is **explicitly deferred** to a
future slice — out of scope here (see Non-Goals).

### Command surface

```
oh update --from <dir> [--dry-run] [--force]
```

- `--from <dir>` (**required** in this slice): a built OpenHarness checkout whose
  `<dir>/.oh/` is the upgrade source. Absent → precondition error (exit 1),
  mirroring init's "templates not found" message and citing #531 deferral.
- `--dry-run`: report every create/overwrite without writing (prefix `[dry-run] `).
- `--force`: proceed even when the version gate would otherwise no-op (already
  up to date) or refuse (downgrade).
- `--help` / `-h`: per-command usage; also surfaced from `oh --help` and
  `oh update --help`.

### Version gate (reuses the existing version field — no new file)

The canonical `.oh`-infrastructure version is **`.oh/cli/package.json#version`**
(currently `0.1.0`) — already the value the CLI banner prints (`__OH_VERSION__`
injected by `build.mjs`). `oh update` reads it from both sides:

- `available` = `<from>/.oh/cli/package.json#version`
- `current`  = `<target>/.oh/cli/package.json#version`

Gate (semver-ish numeric compare, `a.b.c`):

| Relation | Behavior (no `--force`) | Behavior (`--force`) |
|---|---|---|
| `available > current` | **upgrade** — overlay, report `current → available` | upgrade |
| `available === current` | **no-op** — "already up to date (vX)", exit 0 | re-overlay |
| `available < current` | **refuse** — downgrade error, exit 1 | downgrade, overlay |

A dedicated `.oh/VERSION` file was **rejected**: it would duplicate the package
version and invite drift; the existing field is already the single source of the
banner version.

### Overlay mechanics + the safety invariant

`runUpdate` enumerates files under `<from>/.oh/` and writes them under
`<target>/.oh/`, with these rules:

1. **Path-escape guard (the safety invariant):** every resolved destination MUST
   satisfy `dest === targetOh || dest.startsWith(targetOh + sep)` where
   `targetOh = <target>/.oh`. Any path that would escape `<target>/.oh/` throws.
   This is what makes *"project source remains untouched"* hold **by
   construction**: `oh update` is structurally incapable of writing outside `.oh/`.
2. **Skip volatile/runtime paths** so an upgrade never clobbers local build state:
   any path segment equal to `node_modules` or `dist` is skipped (these are
   gitignored build artifacts, regenerated by `oh`'s own `prepare`/`build`).
3. **Overwrite semantics:** existing files are overwritten (the point of an
   upgrade); new files are created with parent dirs as needed; the per-file report
   line is `overwrite <rel>` / `create <rel>` / `skip <rel> (volatile)`.
4. **Preconditions** (all exit 1, message to stderr):
   - `<from>` missing / not a dir / has no `.oh/` → "update source not found …".
   - `<target>/.oh` missing → "not an OpenHarness-equipped repo: no .oh/ at
     <target>. Run `oh init` / vendor `.oh/` first."
   - `<from>` and `<target>` resolve to the same `.oh` → "source and target are
     the same .oh; nothing to update."

### Code shape (self-contained pure-function pattern)

> There is **no** `commands/init.ts` on this base (it ships in PR #334), so this
> module mirrors *no* existing file — it is fully self-contained: a `runX(opts, io)`
> export with no process side-effects, plus a local `walkFiles` and an exported
> `assertDestInTarget` guard.

- New pure module `.oh/cli/src/commands/update.ts` exporting
  `runUpdate(opts: UpdateOptions, io: UpdateIO): Promise<number>` and
  `assertDestInTarget(dest, targetOh, sep): void` — **no** process exit, **no** argv
  parsing, deterministic, fully unit-testable. Interfaces:

  ```ts
  export interface UpdateIO { stdout: (s: string) => void; stderr: (s: string) => void; }
  export interface UpdateOptions {
    targetDir: string;  // equipped repo root (caller resolves; default cwd)
    fromDir: string;    // built OpenHarness checkout (source of the new .oh/)
    force?: boolean;
    dryRun?: boolean;
  }
  ```

- `cli.ts` gains argv parsing for the `update` subcommand (flag extraction,
  `--from` value, default `targetDir = process.cwd()`), calls `runUpdate`, and
  threads the exit code — exactly the `runInit` + cli.ts split init uses.

## Acceptance criteria

- **AC-A** `.oh/cli/src/commands/update.ts` exports `runUpdate(opts, io)` as a pure
  fn (no `process.exit`, no `process.argv`); implements the version gate, overlay,
  path-escape guard, volatile-skip, dry-run, and all preconditions above.
- **AC-B** `cli.ts` dispatches `oh update`, parses `--from/--dry-run/--force`,
  defaults target to cwd, prints `update` in `oh --help` and a dedicated
  `oh update --help`; unknown-flag and missing-`--from` paths exit 1 with a clear
  message. The existing `config`/help/version behavior is unchanged.
- **AC-C** `oh update --from <newer>` overlays `<target>/.oh/` and **never** writes
  outside it (verified by the path-escape test + a project-file-untouched
  assertion); `node_modules`/`dist` segments are skipped.
- **AC-D** Version gate: upgrade proceeds; equal → no-op exit 0; downgrade →
  refuse exit 1; `--force` overrides both. Reported `current → available`.
- **AC-E** `update.test.ts` (vitest, auto-discovered by `vitest.config.ts`'s
  `.oh/cli/**/__tests__/**` include) covers: upgrade overlay, project-file
  untouched, equal no-op, downgrade refuse, `--force`, dry-run writes nothing,
  path-escape throw, each precondition. Full suite stays green.
- **AC-F** After `npm --prefix .oh/cli ci` (REQUIRED — the worktree has no
  `.oh/cli/node_modules`; CI runs this before both gates), `node .oh/cli/build.mjs`
  bundles clean and `tsc --noEmit` (in `.oh/cli`) passes.
- **AC-G** `evals/probes/oh-update.sh` is a 3-state oracle (`# tier:`/`# source:`/
  `# desc:` header, `BASH_SOURCE` self-resolve) asserting the `update` command +
  the `.oh/`-scoped-write guarantee exist; PASS on this tree. One alphabetical
  `evals/RESULTS.md` row added; full probe suite shows 0 regressions.
- **AC-H** `.oh/README.md` documents `oh update` (purpose, `--from` MVP + deferred
  remote-fetch, the `.oh/`-only safety invariant). `CHANGELOG.md` `[Unreleased]`
  gains an entry citing the full mifunedev #531 URL.
- **AC-I** Grep sweep: no stray references claim `oh update` touches project files;
  every new reference points at the `.oh/`-scoped behavior.

## Non-Goals

- **No remote/network source** — `--from <dir>` only; pulling a git ref or npm
  tarball is a future slice.
- **No installed-binary bundling** — same deferral as `oh init` (#334): the
  on-PATH `oh` has no bundled payload until the Dockerfile COPYs one; run from a
  built checkout or pass `--from`.
- **No `oh init` dependency / no edits to init's files** — `commands/init.ts` and
  `.oh/templates/` are PR #334's; this slice neither needs nor touches them.
- **No new `.oh/VERSION` file** — reuse `.oh/cli/package.json#version`.
- **No runtime-path change** — `/home/sandbox/harness` / `OH_PROJECT_ROOT` are
  Phase 1's concern; untouched here.
- **No backup/rollback/interactive-diff UX** — overwrite-in-place is the MVP;
  richer UX is a future slice.
