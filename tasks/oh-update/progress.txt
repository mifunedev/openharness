# Progress ‚Äî oh-update (RFC #531 Phase 3)

## Codebase Patterns

- The CLI lives in `.oh/cli/`. `src/cli.ts` is the entrypoint (a top-level
  `main(process.argv.slice(2)).then(process.exit)` side effect). `build.mjs` bundles
  it via esbuild to `dist/oh.js`, inlining `__OH_VERSION__` from
  `.oh/cli/package.json#version` (currently 0.1.0). Tests import the SOURCE with `.js`
  specifiers (e.g. `await import("../cli.js")`) ‚Äî NodeNext resolution maps `.js` ‚Üí `.ts`;
  vitest transforms TS. Mirror this for `../commands/update.js`.
- There is NO `commands/` dir on this base ‚Äî `oh init` (PR #334, still open) is what
  introduces `commands/init.ts` + `.oh/templates/`. This slice creates
  `commands/update.ts` fresh and does NOT touch or reference init's files. The only
  overlap with #334 is the `cli.ts` subcommand-registration branch (a trivial merge
  union; documented in the PR body).
- `oh init` operates on PROJECT-owned files (harness.yaml, AGENTS.md, .devcontainer/)
  scaffolded FROM `.oh/templates/`. `oh update` operates on the `.oh/` CONTROL PLANE
  itself (cli, scripts, orchestration). Disjoint targets ‚Äî that is why Phase 3 ships
  standalone off origin/development without depending on Phase 2.
- The safety invariant: `runUpdate` resolves `targetOh = <target>/.oh` and writes ONLY
  under it (path-escape guard `dest === targetOh || dest.startsWith(targetOh + sep)`).
  This makes "project source untouched" hold BY CONSTRUCTION ‚Äî the command is
  structurally incapable of writing outside `.oh/`. The test (US-003) and probe (US-004)
  must prove it.
- Version metadata is the EXISTING `.oh/cli/package.json#version` ‚Äî no new VERSION file.
  `runUpdate` reads it from `<from>/.oh/cli/package.json` (available) and
  `<target>/.oh/cli/package.json` (current); numeric dotted compare; refuse downgrade
  without `--force`; no-op when equal without `--force`.
- vitest config (`vitest.config.ts`) includes `.oh/cli/**/__tests__/**/*.test.ts`, so a
  new `update.test.ts` is auto-discovered by CI's Test job. Verify with
  `/home/sandbox/harness/node_modules/.bin/vitest run --root <worktree>` (the worktree
  has no node_modules; resolution walks up to the repo-root node_modules).
- Eval probes are 3-state oracles (exit 0 PASS / 1 REGRESSION / 2 SKIPPED) with a
  `# tier:` / `# source:` / `# desc:` header and `ROOT` resolved via `${BASH_SOURCE[0]}`.
  `evals/RESULTS.md` is a `| probe | tier | last-run (UTC) | status | source |` table,
  one row per probe id, kept alphabetical. The "Eval Probe Regression Gate" CI job
  fails only on a NEW green‚Üíred regression.
