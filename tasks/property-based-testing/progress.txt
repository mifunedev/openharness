### What I did
Created `apps/oh/src/__tests__/cli.property.test.ts` with three property tests:
1. `isHelpFlag` determinism: `expect(isHelpFlag(s)).toBe(isHelpFlag(s))` for all `string | undefined` inputs

3. Combined no-throw: `expect(() => isHelpFlag(s)).not.toThrow()` and `expect(() => isVersionFlag(s)).not.toThrow()` in a single property

The import uses `../cli.js` per NodeNext ESM resolution. `fc.oneof(fc.string(), fc.constant(undefined))` covers the full `string | undefined` signature. Mutual exclusion assertion explicitly excluded per AC.

Key challenge: `cli.ts` has a top-level `main(process.argv.slice(2)).then(process.exit)` call. Vitest intercepts `process.exit` in its fork pool and surfaces it as an unhandled error even when all tests pass. Fixed with `vi.mock("../cli.js", async (importOriginal) => { ... })` â€” the factory stubs `process.exit` to a no-op, awaits the original module import (triggering main()), allows one microtask tick for the promise to resolve, then restores the original. Dynamic `import("../cli.js")` is used in test scope to consume the mocked module.

No fast-check added to `apps/oh/package.json` â€” root hoisting confirmed working (same as US-001 note). `pnpm test` passes: 121 tests across 12 files. Typecheck passes.

### Learnings for future iterations
- `cli.ts`-style modules with top-level `main().then(process.exit)` require `vi.mock` factory hoisting to suppress vitest's `process.exit` interception â€” a plain `vi.spyOn` set up before a static import is too late (ESM hoisting runs imports first).
- The `vi.mock(module, async (importOriginal) => { stub process.exit; await importOriginal(); await microtask; restore; return mod })` pattern is the correct solution for modules with async top-level side effects.
- NodeNext ESM imports in vitest tests do require the `.js` extension in the import path (`../cli.js`) even though source is `.ts`.
